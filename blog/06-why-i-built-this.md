# 为什么我做了一个剪贴板检查工具？一个独立开发者的故事

## 一次说不通的 bug

三年前我在做一个内部工具，功能不复杂：用户从网页上复制一些配置参数，粘贴到输入框，系统解析后执行。测试阶段一切顺利。上线第一天，一位同事报告说粘贴后程序行为异常，解析出来的数据"看着对但就是不对"。

我让他把粘贴的内容截图发过来。肉眼看去，纯文本，干干净净，没有任何异常。我花了一个下午加那条语句打印、用 `JSON.stringify` 逐字对比，最后发现字符串里多了几个不可见的空白字符，来源不明。

那天的调试经历让我第一次认真思考一个问题：当你按下 Ctrl+V，你以为粘贴的是纯文本，但实际上剪贴板里可能藏着完全不同的东西。

## "看起来正常"的错觉

那个 bug 后来查清楚了。同事是从一个富文本编辑器里复制的配置文本。编辑器把内容同时以 `text/plain` 和 `text/html` 两种格式写入了剪贴板。粘贴到我们的输入框时，拿到的是 `text/plain`，但那段纯文本在某些浏览器里被悄悄附加了零宽字符或额外的换行。

这不是个例。后来我在各种场景下反复遇到类似的事：

- 从 Excel 复制数字，粘贴后数字前后多了空格
- 从 Slack 复制代码片段，粘贴到终端后缩进全乱了
- 从 Word 复制文字，粘贴到 Markdown 编辑器后出现莫名其妙的 HTML 标签

每次遇到这种问题，第一反应都是"粘贴端有 bug"。但真正的问题出在更底层：你根本不知道剪贴板里到底装了什么。

## 剪贴板不是你以为的剪贴板

搞清楚这件事之后，我花了些时间研究剪贴板的工作原理。结论很简单但容易被忽视：剪贴板不是一个只装纯文本的容器，它是一个多格式的集装箱。

从浏览器复制一段加粗文字，剪贴板里会同时存储：

- `text/plain`: 纯文本，格式信息全部丢失
- `text/html`: 带标签的 HTML 片段
- 有时还有 `text/uri-list`，如果内容包含链接的话

从 Excel 复制一个单元格，剪贴板里可能同时有 `text/plain`、`text/html`、`text/csv`，甚至 Office 专有的 XML 格式。粘贴时，接收方从这些格式里挑一个自己能处理的。记事本拿 `text/plain`，浏览器拿 `text/html`，各取所需。

这就是为什么"看起来一样的复制操作，粘贴到不同地方效果完全不同"。不是粘贴端的问题，是你从来没看到过剪贴板的全貌。

## 现有工具的空白

知道原因后，我需要一个能看清剪贴板全貌的工具。

macOS 上有一些剪贴板管理器，比如 Paste、Copied。但它们关注的是"管理"，保存历史记录、快速调用、跨设备同步。没人关心剪贴板里到底有几种 MIME 类型，每种格式的原始内容是什么。

Windows 上有 Clipdiary、Ditto，功能类似，同样不关注数据格式检查。

命令行可以用 `pbpaste`（macOS）或 `powershell Get-Clipboard`，但只能拿纯文本，拿不到 HTML 格式，更看不到二进制数据。

浏览器里，有人在 Stack Overflow 上推荐 `clipboardData.types` 加上 `getData()` 手动打印。能用，但每次都要打开 DevTools 写几行代码，体验很差。

我的需求很明确：打开一个页面，粘贴，看到所有格式的原始数据。不需要管理，不需要同步，只需要"检查"。这个需求太小众了，小众到没有现成的工具愿意做。

## 发现上游项目

在 GitHub 上搜了几天，找到了 [evercoder/clipboard-inspector](https://github.com/nickygerritsen/clipboard-inspector)。一个基于 React 的单页应用，功能恰好是我想找的：粘贴或拖拽后，列出所有 MIME 类型并展示内容。

作者做的很好，界面简洁，核心逻辑清晰。但有几个地方不够用：

- 没有导出功能，看到数据后无法保存
- TypeScript 是 loose 模式，类型不够严格
- 没有 CI/CD，部署靠手动
- 没有 Markdown 导出，没法直接丢给 AI 助手分析

我当时正在频繁用 AI 辅助调试。遇到剪贴板问题时，我得先手动截图或复制 DevTools 输出，再粘贴给 AI。如果工具能直接导出一份结构化的 Markdown，把这件事变成一步操作，效率会高很多。

## Fork，改造，踩坑

2024 年底我 fork 了上游项目，开始改造。

第一件事是把 TypeScript 切到 strict 模式。上游代码用了 `any` 和隐式类型推断，改成 `strict: true` + `noUncheckedIndexedAccess` + `exactOptionalPropertyTypes` 之后，一堆类型错误冒出来。光修类型就花了两个晚上。但这步不能省，strict 模式下重构导出功能才有信心。

第二件事是加 Markdown 导出。核心逻辑在 `src/download/markdown.ts`，大概 200 行。要点是把每种 MIME 类型的内容转成 fenced code block，MIME type 推断语言标识，二进制数据只展示元信息。这块写了 13 个单测，覆盖了 fence 转义、table 特殊字符、字节数格式化等边界情况。

第三件事是加 ZIP 导出。有些场景需要保留原始二进制数据（比如粘贴的图片、文件），Markdown 做不到。用 JSZip 打包，目录结构参照 bug report 的常见格式，附上 `metadata.json` 和 `README.txt`。

第四件事是 CI/CD。加了 GitHub Actions workflow，每次 push 到 main 自动跑 typecheck、lint、test、build，然后部署到 GitHub Pages。从此不再手动构建和上传。

整个过程断断续续花了大概三周。工作日晚上和周末抽空写，没有整块时间。有好几次周五晚上改到一半，周一回来忘了上下文，又要重新看代码。独立开发的常态，习惯了。

## Clipboard Inspector

现在的 [Clipboard Inspector](https://fudesign2008.github.io/clipboard-inspector/) 做的事情很简单：

打开页面，粘贴或拖拽，看到剪贴板里的全部数据格式。`text/plain`、`text/html`、`image/png`、各种私有 MIME type，有多少显示多少。HTML 格式同时展示源码和渲染结果，图片直接预览，文件显示名称和大小。

看完了，点一下"导出 Markdown"拿一份结构化的文本文件，可以直接丢给 AI 助手。或者"导出 ZIP"拿一个压缩包，里面有原始数据，适合提交 bug report 或归档。

技术栈是 React 18 + TypeScript strict + esbuild + Vitest。代码在 [GitHub](https://github.com/FuDesign2008/clipboard-inspector)，MIT 协议。

## 从工具到产品的距离

说实话，Clipboard Inspector 目前还是一个工具，不是产品。工具解决具体问题，产品解决一类问题并持续产生价值。

工具阶段该做的事都做了：核心功能稳定，有测试覆盖，有 CI/CD，有文档。但它缺少产品维度的东西，比如用户反馈渠道、使用数据统计、功能迭代规划。

我没有把它做成产品的计划。原因很简单：这是一个需求极小众的工具。大多数开发者一辈子不需要检查剪贴板的原始数据。需要的人，找到它的时候会很开心；不需要的人，永远不会打开它。这种工具做成开源项目比做成产品更合适。

如果你正在做类似的事情，判断"该做成工具还是产品"的一个粗略标准：你的目标用户会不会主动搜索这个功能？如果会，可能是产品机会；如果不会（比如"检查剪贴板数据格式"这种需求，大多数人根本不知道自己需要），那工具就够了。

## 给独立开发者的几句话

做了这些年独立开发，踩了不少坑，分享几个教训。

**从小处着手，但要完整。** Clipboard Inspector 的核心功能就是"粘贴，看到数据"。这个功能从第一天起就是完整的。导出、CI/CD、strict TypeScript 都是后来加的。如果一开始就规划一堆功能，可能到现在还没发布。

**先用自己的工具。** 这个项目最初完全是为了解决我自己的问题。因为每天都在用，所以知道哪里不顺手，哪里需要改进。如果你自己不是目标用户，开发过程中的决策会变得很盲目。

**不要害怕 fork。** 很多独立开发者有一种"从零开始造轮子"的执念。如果有人已经做了 80% 你需要的东西，fork 过来加剩下的 20%，比自己从头写快得多。fork 不是偷懒，是尊重别人的工作并在此基础上创造新价值。

**写测试。** 独立开发通常是一个人，没有同事帮你 review 代码，没有 QA 帮你测功能。测试就是你唯一的同事。13 个单测覆盖 Markdown 导出的边界情况，每次改代码心里都有底。

**接受小众。** 不是每个项目都要有几千个 star、几万个月活用户。解决好一个问题，哪怕受众很小，也是有价值的事。Clipboard Inspector 的 GitHub star 数不多，但每次收到 issue 或邮件说"这工具帮了我大忙"，那个感觉比 star 数字实在得多。
