# D. 一人公司启动检查清单

一人公司的核心约束是时间和资金都有限。在错误的方向上花费一个月，代价远超过多做几次市场调研。这份检查清单把从"开源项目"到"有收入的产品"之间需要完成的事项拆解成可执行的任务，按优先级排列，标注成本和时间估算，方便逐项推进。

优先级定义：

- **P0**：上线前必须完成，否则无法发布或存在法律风险
- **P1**：上线第一个月内完成，直接影响早期用户体验和转化
- **P2**：前三个月内完成，支撑增长和商业化
- **P3**：根据实际需要启动，不必提前投入

## 法律与合规

法律问题不会每天发生，但一旦发生往往代价很高。对于独立开发者而言，合规的核心原则是"花最少的钱覆盖最大的风险面"。早期不需要注册公司，但隐私政策和用户协议必须在产品上线时就位。

- [ ] **P0** 保持 MIT 开源协议不变（$0，无工作量）
- [ ] **P0** 在 GitHub Pages 站点添加 Privacy Policy 页面（$0，约 1 天）
- [ ] **P0** 在 GitHub Pages 站点添加 Terms of Service 页面（$0，约 1 天）
- [ ] **P2** 产生收入后注册个体工商户（¥0-300，1-5 天）
- [ ] **P2** 计划商业化时申请软件著作权（¥300-2500，1-2 个月）
- [ ] **P3** 使用国内服务器时办理 ICP 备案（¥250-5000/年）
- [ ] **P3** 新增服务端功能时考虑 AGPL + 商业双授权模式

> 依赖关系：个体工商户注册应在有收入之后、税务申报之前完成。软著申请周期较长（1-2 个月），如计划在国内分发则应提前启动。

## 基础设施

基础设施的选择决定了后续开发和运维的复杂度。一人公司应该尽可能选择托管服务而非自建，把运维负担降到最低。当前的技术栈（GitHub Pages + esbuild + 静态站点）已经是最简配置，继续保持。

- [ ] **P0** 继续使用 GitHub Pages 托管（$0）
- [ ] **P0** 在 Cloudflare 购买 .dev 域名（~$12/年）
- [ ] **P0** 配置 Cloudflare Email Routing，转发到 Gmail（$0）
- [ ] **P1** 在 index.html 中集成 PostHog 分析脚本（$0，免费 100 万事件/月）
- [ ] **P1** 配置 Better Stack 可用性监控（$0，免费 10 个监控）
- [ ] **P2** 合并 CI/CD 为单一 workflow，加入并发控制
- [ ] **P3** 带宽超限时考虑迁移到 Cloudflare Pages

> 依赖关系：域名购买是其他所有品牌相关工作的前置条件。PostHog 和 Better Stack 都有免费额度，足以覆盖早期流量。

## 产品与开发

产品是核心，但一人公司不能陷入"完美主义陷阱"。MVP 的目标是验证价值假设，不是打磨到完美。TypeScript strict mode 和基本测试覆盖是底线要求，保证代码质量不会成为技术债务。

- [ ] **P0** 按第五部分路线图交付 MVP 功能
- [ ] **P0** 确保 TypeScript strict mode 全部通过
- [ ] **P1** 为关键路径添加单元测试和集成测试
- [ ] **P2** 开发浏览器扩展原型
- [ ] **P2** Core Web Vitals 性能优化

> 依赖关系：浏览器扩展依赖 MVP 核心功能稳定后才能启动。性能优化应在功能基本完成后统一处理。

## 品牌与营销

开源项目不需要品牌，但商业产品需要。品牌建设不是烧钱做广告，而是在目标用户心中建立"这个工具是干什么的、为什么用它"的认知。对于开发者工具而言，名字、logo 和着陆页是三个最基础的触点。

- [ ] **P0** 确定产品名称（Clipboard Inspector / ClipScope / PasteScope）
- [ ] **P0** 制作 logo（Figma 自制，或 Fiverr 外包 $20-100）
- [ ] **P0** 确定色彩体系（Indigo #6366F1 + Slate + Amber 强调色）
- [ ] **P1** 构建着陆页：hero + demo + 定价 + FAQ
- [ ] **P1** 撰写 4-6 篇初始 SEO 文章
- [ ] **P1** 注册 Twitter/X 账号，开始 build in public
- [ ] **P2** 准备 Product Hunt 发布页

> 依赖关系：产品名称和 logo 是着陆页的前置条件。SEO 文章应在着陆页上线后开始发布，为搜索引擎收录争取时间。

## 变现

变现系统的核心是"收钱"这件事本身不能成为瓶颈。LemonSqueezy 处理税务、发票和合规，让开发者专注于产品。订阅流程和 license key 系统需要在收费功能上线前就绪。

- [ ] **P1** 注册 LemonSqueezy 账户（$0，约 30 分钟）
- [ ] **P1** 确定定价层级：Free + Pro $4-6/月 + Team $8-12/用户/月
- [ ] **P2** 实现订阅结账流程
- [ ] **P2** 设置 webhook 处理订阅事件（激活、续费、取消）
- [ ] **P2** 构建 Pro 功能的 license key 系统

> 依赖关系：定价层级确定后才能设置 LemonSqueezy 商品。Webhook handler 是自动化订阅管理的技术基础。

## 运营

运营工具的选择原则是"够用就好"。每个工具都有学习成本，工具越多管理负担越重。早期用免费或低价方案，等收入增长再升级。

- [ ] **P1** 设置 Linear 做项目管理（$8/月）
- [ ] **P1** 设置 Notion 做知识库（$0，免费版足够）
- [ ] **P1** 启用 GitHub Discussions 做用户支持（$0）
- [ ] **P2** 设置 Typefully 做 Twitter 内容排期（$12.50/月）
- [ ] **P2** 设置 1Password 管理 API key 和凭证（$4/月）
- [ ] **P1** 制定每周工作时间分配：开发 40%、营销 25%、支持 15%、运营 10%、学习 10%

> 依赖关系：项目管理和知识库应尽早设置，为后续所有工作提供组织框架。Twitter 排期工具在有内容产出后再开通。

## 社区

社区是独立开发者的增长飞轮。build in public 不只是营销策略，更是一种获取早期反馈、建立信任和寻找同行的机制。在发布产品之前就应开始参与社区。

- [ ] **P1** 创建 GitHub Discussions 分类（功能请求、Bug 报告、Show & Tell）
- [ ] **P1** 加入 Indie Hackers 社区
- [ ] **P1** 寻找 accountability partner，定期同步进展
- [ ] **P2** 制定 build in public 内容日历
- [ ] **P2** 准备发布日检查清单（Product Hunt + Hacker News + Reddit + Twitter）

> 依赖关系：GitHub Discussions 分类设计应在着陆页上线时同步完成。发布日计划需要提前 2-4 周准备。

## 财务规划

财务纪律决定了一人公司能活多久。在收入不确定的早期，控制成本比追求收入更重要。从第一天起就记录收支，建立对现金流的基本感知。

- [ ] **P0** 计算 12 个月个人资金跑道
- [ ] **P0** 设定月度运营预算（$12-133/年，bootstrap 级别）
- [ ] **P1** 从第一天起记录收入和支出
- [ ] **P2** 规划税务申报（个体工商户 + 个人所得税）
- [ ] **P3** MRR 超过 $2K 时评估注册 Wyoming LLC

> 依赖关系：资金跑道计算是所有投入决策的前提。个体工商户注册在产生收入后、需要开发票或申报税务前完成。
