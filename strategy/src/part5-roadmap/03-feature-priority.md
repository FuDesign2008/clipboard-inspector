# 5.3 功能优先级矩阵

功能想法很多，开发资源有限。RICE 评分法提供了一种客观的优先级排序方式，减少主观偏好对决策的干扰。

## RICE 评分方法

RICE 四个维度的定义：

- **Reach（覆盖面）**：预计在给定时间范围内受影响的用户数量。以每季度影响的用户数计量。
- **Impact（影响力）**：对单个用户的影响程度。1 = 最小，2 = 低，3 = 中等，4 = 高，5 = 非常高。
- **Confidence（信心度）**：对 Reach 和 Impact 估计的信心程度。1 = 低信心（猜测），3 = 中等（有一些数据），5 = 高（有充分数据）。
- **Effort（工作量）**：所需的人周数。包括设计、开发、测试和文档。

**RICE 分数 = (Reach x Impact x Confidence) / Effort**

分数越高，优先级越高。

## 优先级梯队分析

### P0：必须做（MVP 核心）

前四项功能的 RICE 分数远高于其他功能，它们构成了 MVP 的核心价值。

**密钥/Token 检测**以 62500 的分数排在首位。它的 Impact 为 5（最高），因为它直接关系到安全问题。Confidence 为 5，因为需求很明确（密钥泄露是真实痛点），实现方式也很确定（纯正则）。Effort 只有 2 人周，投入产出比极高。

**内容类型自动分类**分数紧随其后。它是"剪贴板的 DevTools"定位的核心体验：粘贴内容后自动识别类型并切换到合适的视图。没有这个功能，用户需要手动判断内容类型，体验大打折扣。

**剪贴板历史**的 Confidence 较低（4），因为 `clipboardchange` 事件的浏览器支持仍然有限。但它的 Impact 足够高，且实现成本可控（3 人周），值得在 MVP 中尝试。

**AI 上下文打包导出**的 Confidence 为 3，因为我们假设用户会需要这个功能，但缺乏数据支撑。把它放入 MVP 的目的是验证这个假设。

### P1：应该做（增强体验）

P1 功能的共同特点是：提升体验、实现成本低、覆盖面广。

**主题切换**的 RICE 分数意外地高（100000），因为它的 Reach 很大（所有用户都会受益）且 Effort 极低（0.5 人周）。但它排在 P1 而不是 P0，因为它对核心假设验证的影响较小。开发者不会因为有没有深色模式而决定是否使用这个工具。

**Base64 编解码**和 **JSON 格式化**都是实用性很强的增强功能。它们与内容类型自动分类联动：检测到 Base64 时自动提供解码视图，检测到 JSON 时自动格式化。

**JWT 解码器**虽然只影响部分用户（后端开发者、安全工程师），但对这些用户的 Impact 很高（4），而且实现简单（1.5 人周，JWT 解码就是 Base64 + JSON 解析）。

### P2：可以做（平台扩展）

P2 功能标志着从单页面 Web 工具向平台的扩展。

**浏览器扩展**是走向用户的重要一步。它的 Impact 高（4），但 Effort 也高（4 人周），且 Confidence 较低（3），因为浏览器扩展的用户接受度不确定。

**OCR 文字提取**的需求真实存在（从截图中提取文字），但实现成本中等，且需要引入外部依赖（Tesseract.js，约 4MB）。

**Tauri 桌面应用**的 Impact 最高（5），因为桌面端能解锁 Web 端无法实现的功能（系统级剪贴板访问、常驻后台、全局快捷键）。但 Effort 也最高（12 人周），且 Confidence 较低（2），因为桌面端的用户需求还没有被验证。

### P3：长期方向

P3 功能是长期愿景的一部分，短期内不投入开发资源。

**语义搜索**需要 ML 模型下载（30-80MB），实现成本高，且需要足够大的剪贴板历史数据才有意义。适合在桌面端实现。

**MCP server** 是一个有趣的方向。通过 MCP（Model Context Protocol）server，Claude、Cursor 等 AI 工具可以直接调用 Clipboard Inspector 的能力。但目前 MCP 生态还处于早期，用户需求不确定。

**团队共享和跨设备同步**需要后端基础设施（用户认证、数据存储、实时同步），与"纯静态 Web 应用"的架构理念冲突。只有在产品验证成功、有稳定收入之后才值得考虑。

## 实施建议

RICE 分数是决策的参考，不是唯一标准。实际实施时还需要考虑：

**功能之间的依赖关系**。内容类型自动分类（P0）是 JSON 格式化（P1）和 JWT 解码器（P1）的前置依赖。分类引擎需要先建好，后续的格式化视图才能接入。

**技术风险对冲**。剪贴板历史依赖 `clipboardchange` 事件，如果浏览器支持进展不如预期，需要准备降级方案。

**用户反馈优先**。MVP 上线后，优先根据真实用户反馈调整优先级。如果某个 P1 功能被大量用户请求，可以提前到 P0。如果某个 P0 功能使用率极低，可以重新评估。
